## How This Structure Works in Practice
1. ```infrastructure/``` (The Target)
This is where the actual Terraform code lives. In a real-world scenario, your engineering teams might keep their code in their own repositories, but for a centralized Policy-as-Code lab, keeping it here allows you to test controls across AWS, Azure, and GCP in one place.
2. ```policies/``` (The Rules)
Checkov comes with hundreds of out-of-the-box policies, but eventually, you will need to write custom rules specific to your organization's internal requirements. Storing custom Rego or Checkov YAML files here allows your pipeline to load them automatically during the daily scan.
3. ```evidence/``` (The Machine Artifacts)
This folder is strictly for the raw JSON outputs generated by the GitHub Action. Humans rarely read these files directly; they serve as the cryptographic, version-controlled proof for external auditors.
4. ```governance-vault/``` & ```scripts/``` (The Human View)
Raw JSON is difficult to read and present to leadership. The scripts/ folder holds a lightweight automation script that runs at the end of your pipeline. It parses the JSON files in evidence/ and overwrites the Markdown files inside governance-vault/03-Evidence-Logs/.
Because this folder is entirely Markdown, you can open it directly in local tools to map the live, automated evidence directly to your written ISMS policies without breaking formatting.
